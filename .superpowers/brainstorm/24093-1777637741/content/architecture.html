How is the booking system structured?

This shapes how consumers discover and pay, and how trainers get their money

A

Embedded on trainer profile

The "Book a Session" button lives on [slug].astro. Clicking it opens a booking panel/modal on the same page. Trainer is onboarded to Stripe Connect Express — money goes directly to their bank, platform never touches it.

Pros

  • No extra pages to build
  • Lowest drop-off (consumer stays on profile)
  • Fully automated payouts via Stripe Connect

Cons

  • Profile page gets more complex
  • Modal UX on mobile can be awkward

→ Fastest to ship

B

Dedicated booking page per trainer

/book/[slug] is a separate Astro page optimised for conversion. Profile shows a "Book" button that navigates here. Same Stripe Connect Express payout model.

Pros

  • Clean, distraction-free booking flow
  • Better mobile UX (full-page, not modal)
  • Can add structured data for SEO (BookingAction schema)
  • Shareable URL trainers send to clients directly

Cons

  • One extra page to build
  • One extra navigation hop for consumer

→ Recommended — best UX + shareable

C

External tool (Calendly / Cal.com)

Trainer connects their own Calendly or Cal.com account. Profile shows an embedded widget or a "Book" link that opens their external tool.

Pros

  • Zero infrastructure to build
  • Ship in days

Cons

  • Platform has no booking data — no analytics, no lifecycle emails
  • Not automated — trainer sets up payments separately
  • Trainers pay Calendly on top of TrainedBy
  • Kills the product vision
